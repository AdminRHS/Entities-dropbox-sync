# Анализ требований к приложению Research из звонка руководства

## Суть приложения

Приложение **Research** - это система для управления видео-материалами с функционалом:
- Поиска и синхронизации видео
- Генерации транскриптов из видео
- Таксономического распределения материалов
- Интеграции с Dropbox API
- Работы с базой данных для хранения и отображения данных

---

## Текущая реализация (что уже сделано)

### Основной функционал
1. **Search Key** - добавлен функционал поиска
2. **Синхронизация с Dropbox**:
   - Подключен Dropbox API
   - Возможность синхронизировать файлы из Dropbox
3. **Загрузка данных**:
   - Ручная загрузка из CSV файлов (как было в скриптах)
   - Загрузка из мастер-файла (Master Search Master CSV)
4. **База данных**:
   - Данные из файлов переводятся в БД
   - Пока локальная, потом будет перенесена на корпоративный сервер (Neon или Supabase)
   - Все данные выводятся из БД
5. **CRUD операции**:
   - Добавление
   - Редактирование
   - Удаление записей
6. **Видео Queue**:
   - Добавление видео по ссылке
   - Детальная страница видео
7. **Страница видео**:
   - Копирование ссылки
   - Переход к видео
   - **Генерация транскрипта**
8. **AI интеграция**:
   - Подключен Google Studio AI
   - Доступ через API token
   - Работает через OpenAI API или Google Studio
   - Результат выводится в формате JSON
   - Сформирована структура JSON для транскрипций

### Текущие вопросы
- Вопросы по статусам (в process)
- Выбор правильного шаблона (004 вместо 90)

## Требования руководства

### 1. Система ID и версионирования
- Создать систему ID для всех топиков
- В Dropbox должен быть исходный файл
- Создать индекс-файл
- Внедрить changelog для отслеживания изменений и апдейтов

### 2. Dashboard (Дэшборд)
Требуется создать дэшборд с отображением:
- **Статистика по сотрудникам**: сколько задач/видео у какого сотрудника
- **Статистика по инструментам**: сколько по какому инструменту

### 3. Интеграция с микросервисами
- **Talents** - для получения данных о сотрудниках
- **Libraries** (Лиддер) - для данных по инструментам
- Это будут более поздние этапы

### 4. Версионность API
- **Версия 1**: Простая (текущая реализация)
- **Версия 2**: Более сложная (с интеграциями)
- **Версия 3+**: Дальнейшие усложнения
- На каждую версию нужен план разработки через pseudo-code

### 5. Виджет выбора
- Создать переиспользуемый элемент/виджет для выбора вариантов
- Будет использоваться на множестве страниц
- Принцип: "выбери, какой вариант более подходящий/правильный"

### 6. Документация
Вся документация должна быть сложена в папку `documentation/`:
- Def/Deck/System - структура на выбор
- Полный контекст
- Примеры (examples)
- Следовать принципу "Claude Code"
